## 1\. 文档目的与使用方式

本文件是一份量化系统需求文档与软件架构说明，用于明确系统目标、输入输出、操作流程、软件分层、数据 contract 和实现边界。本文档强调的是系统本身应当具备的能力与约束，而不是面向某个特定工具的提示词。

本框架围绕统一中间层设计展开。核心中间对象是 active_session_index、panel_base、target_block、panel_labeled。模型、视图、训练与回测都建立在这些对象之上。

实现要求如下。

- 优先遵守 schema contract、模块职责和配置契约。
- 对未明确要求的部分，选择最简单、最稳、最容易验证的方案。
- 不要为了局部性能优化破坏统一中间层。
- 任何会影响 panel_base 与 panel_labeled contract 的修改，都必须先更新 validator 与测试。

## 2\. 项目背景、目标与非目标

### 2.1 项目背景

用户需要搭建一个 Python 量化研究与训练框架。这个框架必须能够从原始 OHLCV 与因子数据出发，构建标准化资产表，生成统一 panel_base，按 recipe 构建 target_block，再拼成 panel_labeled，供不同模型训练和回测使用。

该框架受 qlib 的 panel 思想启发，但不要求复制 qlib 的全部实现。重点是继承“统一索引、统一底板、标签外挂、视图分叉”的思想，并将其做成更适合用户自己扩展的工程结构。

### 2.2 核心目标

- 稳定生成统一 schema 的 panel_base 与 panel_labeled。
- 支持复杂 label 系统，并将 target 作为一级模块独立管理。
- 支持通用模型与强约束模型共存，并通过 model-label contract 防止错误配置。
- 支持同一套中间数据服务于 LightGBM、DNN、LSTM 以及未来更多模型。
- 支持脚本化 end-to-end 流程，包括 assets、panel、target、train、predict、backtest。
- 支持后续扩展为多任务训练、多 seed 训练、更复杂的回测与报告系统。
- 对行情表与因子表的缺失、不对齐、单边存在等情况有显式表达和可追踪处理。

### 2.3 非目标

- 首版不追求高频、分钟级或事件驱动框架。
- 首版不追求分布式训练、超大规模并行计算或复杂调度系统。
- 首版不实现线上实时交易系统，只实现研究、训练、预测与离线回测。
- 首版不要求兼容所有券商或数据供应商格式，只需通过 sources 层留出扩展点。

## 3\. 核心设计原则

| 原则  | 解释  |
| --- | --- |
| 统一索引优先 | 所有核心中间表统一使用逻辑主键 \[date, sid\]。 |
| 统一底板优先 | panel_base 只表达已知事实，不表达监督目标。 |
| 标签外挂优先 | target_block 独立生成，再与 panel_base 拼出 panel_labeled。 |
| view 分叉优先 | 不同模型在 views 层分叉，不在 panel_base 层分叉。 |
| 契约先于实现 | schema contract 与 model-label contract 必须先定义，再写代码。 |
| 错误尽量前置 | 配置错误、schema 错误、语义错误要在训练前被拦截。 |
| 主输入固定 | 首版唯一标准输入是每日每股 OHLCV 表与每日每股因子表。 |
| 行由索引决定 | panel_base 的样本行空间由 active_session_index 决定，而不是由 OHLCV 或 factor 任一表决定。 |
| 来源可追踪 | 对 market only、factor only、no source record 等情况必须显式建模。 |
| MVP 先通路再扩展 | 先跑通 panel_base、单 label、LGB、回测，再加多任务与深度模型。 |

## 4\. 面向使用者的操作型用户故事与操作流程

这一节不是偏抽象的“业务用户故事”，而是直接告诉系统使用者：我需要准备什么输入数据，我应该按什么顺序运行，我怎样新增一个 label，我怎样新增一个 model，以及当配置写错时系统应该怎样报错。

系统实现必须优先满足这里定义的操作流程，而不是只满足模块名字存在。

### 4.1 使用者首先要准备什么输入数据

首版系统唯一标准输入是两张日频、逐股的数据表，并且系统内部统一使用长表主键 \[date, sid\]。如果用户原始数据是宽表，必须先在 sources 或 assets 阶段转换成长表，再进入 panel 层。

输入表一：每日每股 OHLCV 表

|     |     |     |
| --- | --- | --- |
| **字段** | **是否必需** | **说明** |
| date | 是   | 交易日，推荐使用 YYYY-MM-DD。 |
| sid | 是   | 系统内部股票主键。首版建议直接使用稳定证券代码。 |
| symbol | 建议  | 展示用证券代码，允许与 sid 相同。 |
| open | 是   | 开盘价。 |
| high | 是   | 最高价。 |
| low | 是   | 最低价。 |
| close | 是   | 收盘价。 |
| volume | 是   | 成交量。 |

OHLCV 表最低约束如下。

主键层面必须满足 (date, sid) 唯一。

列名进入系统后会被标准化为 market.open、market.high、market.low、market.close、market.volume、market.amount。

首版默认这是日频口径的数据，复权方式由用户在配置中说明，框架不替用户决定。

如果某个 (date, sid) 在 OHLCV 表中不存在，不代表样本不存在，只代表 status.has_market_record=False。

输入表二：每日每股因子表

|     |     |     |
| --- | --- | --- |
| **字段** | **是否必需** | **说明** |
| date | 是   | 交易日。 |
| sid | 是   | 系统内部股票主键。 |
| factor_\* 或原始因子列 | 是   | 一个或多个数值型因子列。 |

因子表最低约束如下。

主键层面必须满足 (date, sid) 唯一。

进入系统后，所有因子列统一重命名到 feature.\* 命名空间。

因子表允许某些 (date, sid) 完全无记录，也允许有记录但部分字段为空。

如果某个 (date, sid) 在因子表中不存在，应当被识别为 status.has_factor_record=False 与 status.factor_row_missing=True。

因子表中不要有missing flag这种列，因子表允许有NAN值

|     |     |     |     |
| --- | --- | --- | --- |
| **步骤** | **用户要做什么** | **系统应该做什么** | **主要产出** |
| 第 1 步 | 准备原始 OHLCV 表与原始因子表，并放入约定目录 | 校验文件存在、列名和 (date, sid) 唯一性 | data/raw/... |

| 第 4 步 | 检查 panel_base 质量 | 输出 has_market_record、has_factor_record、sample_state 的分布统计 | 质量报告或统计表 |
| 第 5 步 | 选择一个 label recipe 并运行 build_targets | 生成 target_block，输出 label.\*、label_valid.\*、label_reason.\* | data/processed/targets/&lt;target_name&gt;/... |

| 第 8 步 | 运行 predict 或 backtest | 基于 prediction 与市场状态过滤生成结果 | 预测文件、回测报告 |

下面的“用户故事”只写系统使用动作，不写泛泛的业务背景。系统实现必须确保用户可以按这些动作直接使用框架。

|     |     |     |     |     |
| --- | --- | --- | --- | --- |
| **ID** | **场景** | **用户怎么操作** | **系统必须支持什么** | **验收标准** |
| OPS-01 | 接入原始数据 | 把每日每股 OHLCV 表和每日每股因子表放到 data/raw，并在配置里写路径与列名映射 | 读取文件、做列名映射、校验主键唯一性 | 输入校验脚本能输出通过报告或明确错误信息 |
| OPS-02 | 生成标准资产层 | 运行 build_assets | 生成 instrument_master、trading_calendar、daily_bars、factor_values | 产物可独立落盘，且字段符合约定 schema |
| OPS-03 | 构建统一中间层 | 运行 build_panel_base | 以 active_session_index 为主表，分别 left join 行情与因子 | panel_base 中能看到 market only、factor only 等状态 |
| OPS-04 | 检查中间层质量 | 查看 sample_state 分布、缺失比例和主键覆盖率 | 输出质量报告与统计表 | 用户能快速定位异常日期、异常股票和异常状态 |
| OPS-05 | 新增一个 label | 对收益类 target：在 registry 中注册新公式即可（如 `register_target("return_o1c5", ReturnRecipe, ...)`）；对全新类型：新增 recipe 文件并注册 | 自动生成 target_block；config 中只需引用名称，registry 管理默认参数 | 无需改 trainer 主流程；收益类 target 无需新建文件，只需一行注册 |
| OPS-06 | 使用通用模型训练任意 label | 在 config 中选择 ANY_SINGLE 或 ANY_MULTI 模型，并填入 labels | 在 preflight 阶段检查 label 数量与类型 | 合法配置能训练，非法配置会在进入模型前报错 |
| OPS-07 | 用户可以新增通用模型 | 用户需要写好 model 与 view，并声明 contract | 允许该模型复用不同 label 配置 | 更换 label 后仍可复用同一模型代码训练 |
| OPS-08 | 用户可以新增强约束模型 | 新增多任务模型，并声明 role 或 exact label 契约 | 根据 label_roles 或 exact_labels 做校验 | 配错 label 时在训练前报错 |
| OPS-09 | 运行完整训练实验 | 写实验 config，指定数据区间、label、model、view、训练参数，并可以指定参与训练的因子有哪些 | 执行数据读取、view 构建、训练、验证和保存 | data/training_result/&lt;exp_name&gt; 下产出 checkpoint、metrics、predictions |
| OPS-10 | 做预测和回测 | 选择已训练模型运行 predict 与 backtest | 基于 pred.\*、market.\*、status.\* 完成过滤与模拟 | 产出可复现的预测文件与回测报告 |
| OPS-11 | 调试配置错误 | 把不兼容的 label 填给强约束模型 | 不能拖到模型内部炸，必须在 preflight 阶段直接报错 | 错误信息明确指出哪个模型需要什么、当前给了什么 |
| OPS-12 | 逐步扩展框架 | 后续新增更多 label、model 和 backtest 策略 | 允许局部扩展，不需要修改 panel_base 主干 contract | 新增模块时不破坏既有实验与中间层 schema |
| OPS-13 | 测试集评估前可选 refit | 在 config 中设置 `refit_before_test: true`，训练完成并在 valid 上完成模型选择后，系统自动用 train+valid 数据 refit 模型，再在 test 上评估 | 支持 split → train on train → select on valid → (optional) refit on train+valid → evaluate on test 的完整流程；refit 使用选择阶段冻结的超参数和迭代次数；test 集在整个选择过程中不参与任何决策 | refit 前后模型分别保存；test 评估结果记录模型来源（refit 或 selection）；不开启 refit 时行为与原来完全一致 |
| OPS-14 | 断点续跑 pipeline | 运行 `run_pipeline.py`，失败后再次运行相同命令 | 自动检测已完成步骤并跳过；config 变化时自动从头开始；支持 `--restart` 强制重新开始、`--start <step>` 手动指定起始步骤、`--dry-run` 预览 | 检查点文件记录 config hash 和完成状态；不需要用户手动管理断点 |
| OPS-15 | 选择回归损失类型 | 在 config 中设置 `use_lambdarank: true` 切换到 LambdaRank NDCG 排序损失；`false` 使用默认 MSE | 支持 MSE 和 LambdaRank 两种回归损失；LambdaRank 参数（k、sigma、bins）可配置 | 切换损失类型只需改 config，无需改代码 |
| OPS-16 | 两级因子筛选 | 在 `model.view` 中配置 include/exclude_pattern 控制 memmap 中的因子；在 `training` 中配置 include/exclude_pattern 动态过滤 | 构建时筛选影响 memmap 内容（改了需重建 view）；训练时筛选动态生效（改了无需重建 view） | 两级筛选独立生效，互不干扰 |

### 4.4 使用者怎样新增一个 label

用户新增 label 的标准动作应该固定，系统设计应围绕这个动作流来组织目录和接口。

**方式一：利用已有 Recipe 类注册新 target（推荐）**

系统采用 registry 模式管理 target。每个 target 由 recipe 类 + TargetSpec（含默认参数）组成，注册后 config 只需引用名称即可。

例如，收益类 target 统一使用 ReturnRecipe，支持量化标准 price 公式表示法：`{o|c}{day}{o|c}{day}`，其中 o=开盘价、c=收盘价、数字=交易日偏移。例如 `c0c1` 表示今日收盘到明日收盘，`o1c2` 表示明日开盘到后日收盘。

注册一个新的收益 target 只需在 engine/targets/registry.py 中添加一条注册：

```python
register_target("return_o1c5", ReturnRecipe, TargetSpec(
    name="return_o1c5", task_type="regression", family="return",
    horizon=5, dtype="float32", params={"formula": "o1c5"},
))
```

已预注册的收益 target 包括：

| 名称 | 公式 | 含义 |
| --- | --- | --- |
| return_c0c1 | c0c1 | 今日收盘 → 明日收盘 |
| return_o1c2 | o1c2 | 明日开盘 → 后日收盘 |
| return_5d | c0c5 | 今日收盘 → 5日后收盘 |
| return_20d | c0c20 | 今日收盘 → 20日后收盘 |

在 config 中使用时只需引用名称，无需重复填写参数（registry 已有默认值）：

```yaml
targets:
  - name: "return_c0c1"
  - name: "momentum_cls"
```

也可以在 config 中覆盖 registry 的默认参数（engine.py 会自动合并 spec.params 与 config 覆盖值）：

```yaml
targets:
  - name: "return_c0c1"
    params:
      open_col: "market.open_adj"
```

**方式二：新增全新的 Recipe 类**

如果需要全新的 label 逻辑（如动量分类、波动率等），则：

1. 在 engine/targets/recipes/ 下新增文件，继承 BaseTargetRecipe，实现 compute() 方法。
2. 在 engine/targets/registry.py 中注册该 recipe 类与 TargetSpec。
3. 在 config 中引用注册名称。

系统约束如下。

新 label 的输出必须包含 label.&lt;target_name&gt;、label_valid.&lt;target_name&gt;、label_reason.&lt;target_name&gt;。

recipe 不允许直接改写 panel_base。

TargetSpec 至少要声明 name、task_type、family、horizon、dtype。

如果 recipe 所需列在 panel_base 中不存在，系统必须在 target 构建阶段报错。

验收标准：新增 label 后，无需修改 trainer 主流程；新 label 能单独构建出 target 文件；新 label 能被通用模型或强约束模型按 contract 正确消费。

### 4.5 使用者怎样新增一个 model

系统采用 **Model Triple** 架构：每个模型由三个组件组成，按 `impl/<model_name>/` 目录分组。

| 组件 | 位置 | 职责 |
| --- | --- | --- |
| Model | engine/models/impl/\<name\>/model.py | 网络结构定义（forward、save、load） |
| ViewBuilder | engine/views/impl/\<name\>/view.py | 把 panel_labeled 转为模型输入（如 memmap） |
| Trainer | engine/training/impl/\<name\>/trainer.py | 模型专属训练循环、损失计算 |

三个组件各自继承 base 类（BaseModel、BaseViewBuilder、BaseTrainer），在 engine/models/registry.py 中作为一个 triple 统一注册。

**新增模型的标准步骤：**

1. 在 engine/models/impl/\<name\>/model.py 中实现模型类，继承 BaseModel。
2. 在 engine/views/impl/\<name\>/view.py 中实现 ViewBuilder，继承 BaseViewBuilder。
3. 在 engine/training/impl/\<name\>/trainer.py 中实现 Trainer，继承 BaseTrainer。BaseTrainer 提供共享工具（set_seed、resolve_device、setup_experiment、run_preflight、load_view_meta、build_split 等），子类只需关注训练循环本身。
4. 在 engine/models/registry.py 中注册三元组。
5. 在 config 中指定 model.name 即可使用。

**通用模型 vs 强约束模型**

新增 model 时，必须先区分它是通用模型还是强约束模型。

通用模型声明 ModelLabelContract 为 ANY_SINGLE 或 ANY_MULTI，不写死具体 label 名称。

强约束模型声明 contract 为 EXACT、ROLE_BASED 或 CUSTOM。推荐使用 ROLE_BASED，模型依赖角色而非具体 label 名称。例如当前的 lstm_mtl 模型声明 label_roles: {regression: "return_c0c1", classification: "momentum_cls"}，在 config 中可以映射到不同的 target。

如果用户在 config 中填错了 label、role、数量或类型，必须在 preflight 阶段报错。错误信息要可读，不允许只抛出底层 KeyError。

验收标准：新增 model 时，panel_base 与 target_block 的 schema 不需要改；对通用模型，更换 label 只需改 config；对强约束模型，错误配置会在训练前被拦截。

### 4.6 一个典型实验应该怎样配置

下面给出当前已实现的配置方式和扩展示例。配置加载、校验与训练流程必须支持这些写法。

**当前实现：ROLE_BASED 多任务模型（lstm_mtl）**

```yaml
data:
  raw_ohlcv_dir: "C:/AI_STOCK/dataset/rq_ohlcv_yearly_parquet2005-2025"
  raw_factor_dir: "C:/AI_STOCK/dataset/alpha158_plus_fund_yearly_parquet"
  processed_dir: "data/processed"

targets:
  - name: "return_c0c1"        # registry 已有默认参数，无需重复配置
  - name: "momentum_cls"

model:
  name: "lstm_mtl"
  params:
    embed_dim: 128
    hidden_size: 128
    num_layers: 2
    dropout: 0.2
    num_classes: 5
  label_roles:
    regression: "return_c0c1"
    classification: "momentum_cls"
  view:
    lookback: 60
    k: 512

training:
  seed: 42
  split:
    train_start: "2019-01-01"
    train_end:   "2022-01-01"
    valid_start: "2022-01-01"
    valid_end:   "2024-01-01"
    test_start:  "2024-01-01"
    test_end:    "2025-12-15"
  epochs: 100
  lr: 2.0e-4
  batch_size: 4
  patience: 10
  amp: true

evaluation:
  run_test: true
  refit_before_test: false
```

解释：lstm_mtl 是 ROLE_BASED 多任务模型，需要 regression 和 classification 两个角色。config 中 label_roles 把角色映射到具体 target 名称。所有训练超参数统一读取时使用 `.get(key, default)` 模式，未使用的参数不会报错。

**扩展示例：通用单任务模型**

```yaml
targets:
  - name: "return_5d"
model:
  name: "lgb"
```

解释：lgb 属于通用单任务模型（ANY_SINGLE），只需一个合法 label。更换 label 只需改 config。

**扩展示例：通用多任务模型**

```yaml
targets:
  - name: "return_c0c1"
  - name: "return_5d"
  - name: "return_20d"
model:
  name: "generic_mtl"
```

解释：模型 contract 是 ANY_MULTI，接收任意多个合法 label。

### 4.7 当用户配错时，系统应该怎样响应

|     |     |     |
| --- | --- | --- |
| **错误类型** | **典型触发方式** | **系统应该怎样报错** |
| 输入数据错误 | 缺少 date 或 sid 列；主键不唯一 | 在 sources 或 assets 阶段直接报错，并说明缺少什么列或哪张表主键重复 |
| 中间层 schema 错误 | panel_base 缺少 recipe 所需列 | 在 target 构建阶段报错，并指出缺少哪些列 |
| label 配置错误 | 通用单任务模型却给了多个 label | 在 preflight 阶段报错，指出模型允许的 label 数量 |
| 强约束模型配置错误 | ROLE_BASED 模型缺少某个 role | 在 preflight 阶段报错，指出缺少哪个 role |
| 数据对齐异常 | 大量 MARKET_ONLY、FACTOR_ONLY 或 NO_SOURCE_RECORD | 在质量报告中输出统计，并允许用户设置阈值报警 |
| 视图构建错误 | 某些 label 全部无效，无法构建训练样本 | 在 view builder 或 trainer 开始前报错，提示没有可训练样本 |

### 4.8 本节的直接实现要求

必须有明确的数据入口配置，允许用户指定 OHLCV 表、因子表路径和列名映射。

必须有脚本或命令可以单独运行 build_assets、build_panel_base、build_targets、assemble_panel_labeled、train、predict、backtest。

必须允许用户通过“新增 recipe 并注册”的方式扩展 label。

必须允许用户通过“新增 model 和 view 并注册”的方式扩展模型。

必须有 preflight validation，把大部分配置错误拦在训练开始前。

必须给出清晰、可定位、对使用者友好的报错信息。

## 5\. 领域模型与核心对象

### 5.1 active_session_index

定义：panel_base 与 panel_labeled 的逻辑行主键集合。它回答“哪些样本应该存在”。

- 逻辑主键：\[date, sid\]
- 来源：instrument_master 与 trading_calendar
- 包含：上市后到退市前的交易日样本，即使停牌、缺 bar，也应保留该行
- 不包含：尚未上市或已退市样本
- 不是 status 列，不放在 status.\* 中

### 5.2 panel_base

定义：统一底板，只表达某个 date、sid 在当下已知的事实。

panel_base 由 active_session_index 驱动，再把 daily_bars、factor_values、instrument_master、status_intervals 通过 **left join** 的方式拼接进来。任何情况下都不允许让 OHLCV 表或因子表单独决定行空间。

推荐 schema 如下。

index = \["date", "sid"\]  
<br/>columns = \[  
"meta.symbol",  
"meta.list_date",  
<br/>"status.is_listed",  
"status.is_suspended",  
<br/>"status.has_market_record",  
"status.has_factor_record",  
<br/>"status.market_state",  
"status.factor_state",  
<br/>"status.bar_missing",  
"status.factor_row_missing",  
"status.factor_missing_ratio",  
"status.feature_all_missing",  
<br/>"status.sample_state",  
"status.sample_usable_for_feature",  
"status.sample_usable_for_label",  
"status.sample_usable_for_training",  
<br/>"market.open",  
"market.high",  
"market.low",  
"market.close",  
"market.volume",  
"market.amount",  
<br/>"feature.\*",  
\]

这些字段中，新增的关键状态列必须实现。

- status.has_market_record：原始或清洗后的 OHLCV 表中是否存在该 (date, sid) 行。
- status.has_factor_record：因子表中是否存在该 (date, sid) 行。
- status.factor_row_missing：因子源是否整行缺失，而不是仅仅某几个特征为空。
- status.market_state：建议枚举为 OK、MISSING、SUSPENDED。
- status.factor_state：建议枚举为 OK、ROW_MISSING、PARTIAL_MISSING、ALL_MISSING。
- status.sample_usable_for_feature：该行是否可以进入特征构建或 view 层。
- status.sample_usable_for_label：该行是否适合被 target recipe 用于生成当前监督目标。
- status.sample_usable_for_training：该行是否适合直接进入模型训练。

### 5.3 源表存在性四种组合

panel_base 必须显式支持下面四种情况。

| 情况  | has_market_record | has_factor_record | 典型 sample_state | 说明  |
| --- | --- | --- | --- | --- |
| OHLCV 有，factor 有 | True | True | NORMAL | 标准样本。 |
| OHLCV 有，factor 无 | True | False | MARKET_ONLY | 行情存在，但因子源没有该行。 |
| OHLCV 无，factor 有 | False | True | FACTOR_ONLY | 因子存在，但行情源没有该行。 |
| OHLCV 无，factor 无 | False | False | NO_SOURCE_RECORD | 理论上应存在的样本，在两个源表里都没有记录。 |

### 5.4 sample_state 的最低枚举要求

系统至少支持以下枚举值。

SAMPLE_STATE = \[
"NORMAL",
"SUSPENDED",
"MARKET_ONLY",
"FACTOR_ONLY",
"NO_SOURCE_RECORD",
"PARTIAL_FACTOR_MISSING",
"INVALID_BASE_SAMPLE",
\]

说明如下。

- NORMAL：两个源表都可用，且状态正常。
- SUSPENDED：停牌样本（上市窗口内但 OHLCV 整行缺失）。
- MARKET_ONLY：行情存在，因子整行缺失。
- FACTOR_ONLY：因子存在，行情整行缺失（通常与 SUSPENDED 重叠，优先标记为 SUSPENDED）。
- NO_SOURCE_RECORD：两个源表都无记录（停牌且无因子）。
- PARTIAL_FACTOR_MISSING：因子行存在，但部分因子字段缺失。
- INVALID_BASE_SAMPLE：从底层事实看就是异常样本。

### 5.5 target_block

定义：统一 label 输出块，只表达某个 target 定义下的监督目标及其有效性。

index = \["date", "sid"\]  
<br/>columns = \[  
"label.&lt;target_name&gt;",  
"label_valid.&lt;target_name&gt;",  
"label_reason.&lt;target_name&gt;",  
\]

### 5.6 panel_labeled

定义：最终统一训练表，由 panel_base 与 target_block 拼接得到。

panel_labeled = concat(panel_base, target_block, axis=1)

panel_labeled 是训练、视图构建与部分回测输入的核心来源，但不意味着系统要永久维护一张超级大宽表。中间产物可以按 target 分目录落盘。

## 6\. 软件架构总览

系统采用分层架构。数据从 sources 层进入，在 assets 层标准化，在 panel 层形成统一中间层，在 targets 层生成监督目标，在 views 层适配模型输入，在 models 层训练与预测，在 training 与 backtest 层完成实验与评估。

### 6.1 数据流

raw_ohlcv + raw_factors  
\-> assets  
\-> instrument_master / trading_calendar / daily_bars / factor_values / status_intervals  
\-> active_session_index  
\-> panel_base  
\-> target_block  
\-> panel_labeled  
\-> model_view  
\-> model  
\-> trainer  
\-> prediction  
\-> backtest

### 6.2 分层说明

| 层   | 主要职责 | 边界  |
| --- | --- | --- |
| sources | 读取原始输入并标准化列名 | 不做训练逻辑，不做 label 逻辑。 |
| assets | 生成底层资产表 | 把原始数据整理成后续统一使用的标准数据集。 |
| panel | 构建 active_session_index、status、panel_base、panel_labeled | 这是系统核心。 |
| targets | 用 recipe 生成 target_block | 支持简单标签与复杂标签。 |
| views | 把 panel_labeled 转成模型输入（如 memmap）。每个模型在 views/impl/\<name\>/ 下有专属 ViewBuilder。 | 按模型分叉，但保留统一接口（BaseViewBuilder）。 |
| models | 封装模型网络结构（forward、save、load）。每个模型在 models/impl/\<name\>/ 下有专属实现。 | 不直接负责数据清洗。通过 registry 注册三元组（Model + ViewBuilder + Trainer）。 |
| training | 统一切分、训练、模型选择、可选 refit、测试评估。BaseTrainer 提供共享工具，每个模型在 training/impl/\<name\>/ 下有专属 Trainer。完整流程为：split → train on train → select on valid → (optional) refit on train+valid → evaluate on test | 连接 view 与 model。支持 MSE/LambdaRank + CE 多任务损失。selection 阶段冻结超参数与最佳迭代次数；test 集在整个选择过程中不参与任何决策。 |
| backtest | 使用 prediction 与 market、status 列回测 | 输出收益、指标与报告。 |
| schema/core/io | 约束、注册、路径、校验、常量、异常 | 提供公共基础设施。 |

### 6.3 panel 构建核心规则

panel 层必须严格遵守如下规则。

1.  使用 instrument_master 与 trading_calendar 生成 active_session_index。
2.  对 daily_bars 与 factor_values 做主键级去重与标准化。
3.  以 active_session_index 为主表，将 daily_bars 与 factor_values 分别 left join 进来。
4.  根据 join 结果生成 status.has_market_record 与 status.has_factor_record。
5.  进一步生成 market_state、factor_state、sample_state 与 usable flags。
6.  任何 target recipe 或 model view 都不得反向修改 panel_base 的 schema。

推荐伪代码如下。

panel_base = (  
active_session_index  
.left_join(daily_bars, on=\["date", "sid"\])  
.left_join(factor_values, on=\["date", "sid"\])  
.left_join(instrument_master_meta, on=\["sid"\])  
.left_join(status_columns, on=\["date", "sid"\])  
)

## 8\. 数据 schema 与 contract

### 8.1 命名规则

- 核心索引列名统一为 date 与 sid。
- 元数据列使用 meta.\* 前缀。
- 状态列使用 status.\* 前缀。
- 行情列使用 market.\* 前缀。
- 因子列使用 feature.\* 前缀。
- 标签列使用 label.\*、label_valid.\*、label_reason.\* 前缀。
- 预测输出建议使用 pred.\* 前缀。

### 8.2 输入表 contract

| 对象  | 最低字段要求 | 备注  |
| --- | --- | --- |
| daily_stock_ohlcv | date, sid 或 symbol, open, high, low, close, volume, amount | 可以从 symbol 推导 sid。 |
| daily_stock_factors | date, sid 或 symbol, factor_\* | 最终需标准化为 feature.\*。 |

### 8.3 关键中间表 contract

| 对象  | 最低字段要求 | 备注  |
| --- | --- | --- |
| instrument_master | sid, symbol, list_date, delist_date | sid 唯一，日期合法。 |
| trading_calendar | date, is_open | date 单调且无重复。 |
| daily_bars | date, sid, market.open, market.high, market.low, market.close, market.volume | 允许部分缺失，但必须能与 active index 左连接。 |
| factor_values | date, sid, feature.\* | 列名前缀统一，允许 feature 缺失。 |
| panel_base | index=\[date,sid\] 与 meta/status/market/feature 列组 | 不能包含 label.\*。 |
| target_block | index=\[date,sid\] 与 label/label_valid/label_reason | 一个 target block 对应一个 target recipe。 |
| panel_labeled | panel_base 全部列 + target_block 全部列 | 索引与 panel_base 对齐。 |

### 8.4 panel_base 的最低状态字段要求

panel_base 至少必须包含以下状态字段。

REQUIRED_PANEL_BASE_STATUS_COLUMNS = \[
"status.is_listed",
"status.is_suspended",
"status.has_market_record",
"status.has_factor_record",
"status.market_state",
"status.factor_state",
"status.bar_missing",
"status.factor_row_missing",
"status.factor_missing_ratio",
"status.feature_all_missing",
"status.sample_state",
"status.sample_usable_for_feature",
\]

这 12 个字段都遵循一个原则：它们要么是底层业务事实，要么是底层数据事实，要么是基于底层事实得到的汇总状态。以下逐个说明。

#### 8.4.1 status.is_listed

表示这只股票在当前 date 是否处于上市状态。

- True：已上市且未退市。
- False：未上市或已退市。

它是业务事实，不表示数据是否到达。在 active_session_index 的行空间内，is_listed 恒为 True（因为 active_session_index 只包含上市窗口内的样本）。之所以显式保留，是为了让下游逻辑可以直接引用，无需重新推导。

#### 8.4.2 status.is_suspended

表示这只股票在当前 date 是否停牌。

- True：停牌。
- False：未停牌。

它是业务事实。在本系统使用的 OHLCV 数据源中，停牌股票在停牌日不会出现在 OHLCV 表中（整行缺失），因此判定规则为：is_suspended = is_listed AND NOT has_market_record。

它和 bar_missing 不同：is_suspended 是业务状态（为什么没数据），bar_missing 是数据可用性（数据能不能用）。

#### 8.4.3 status.has_market_record

表示原始 OHLCV 表中，是否存在这条 (date, sid) 记录。

- True：OHLCV 表里有这一行。
- False：OHLCV 表里没有这一行。

注意它只表示整行是否存在，不表示字段是否完整。例如：有一行但 close 为空，仍然算 True；根本没有这一行，才是 False。

#### 8.4.4 status.has_factor_record

表示原始 factor 表中，是否存在这条 (date, sid) 记录。

- True：factor 表里有这一行。
- False：factor 表里没有这一行。

这个字段很重要，因为它能显式区分：OHLCV 有而 factor 无，以及 OHLCV 无而 factor 有。

#### 8.4.5 status.market_state

当前样本在市场数据侧的综合状态。枚举值如下。

- OK：市场数据正常可用（有记录且关键字段完整）。
- SUSPENDED：业务上停牌（无 OHLCV 记录）。
- PARTIAL_MISSING：有记录，但关键字段（如 close）缺失。
- MISSING：OHLCV 整行缺失（保留值，当前数据源下与 SUSPENDED 等价）。

它是汇总状态列，便于调试和过滤。

#### 8.4.6 status.factor_state

当前样本在因子数据侧的综合状态。枚举值如下。

- OK：因子存在且足够完整（factor_missing_ratio = 0）。
- ROW_MISSING：factor 表没有这一行。
- PARTIAL_MISSING：有这一行，但部分 feature 缺失。
- ALL_MISSING：有这一行，但全部 feature.\* 为空。

它也是汇总状态列。

#### 8.4.7 status.bar_missing

当前样本是否缺少可用的市场 bar 数据。

- True：bar 不可用。
- False：bar 可用。

这里的 bar 指 market.open、market.high、market.low、market.close、market.volume、market.amount。

它和 has_market_record 的区别是：

- has_market_record=False：整行没来。
- bar_missing=True：这条市场数据不能正常用。

因此可能出现 has_market_record=True 但 bar_missing=True，即有记录但数据损坏。

#### 8.4.8 status.factor_row_missing

factor 表中，这条 (date, sid) 是否整行缺失。

- True：整行不存在。
- False：整行存在。

它本质上等价于 NOT has_factor_record。之所以保留，是因为在写缺失规则和做统计时更顺手、更直观。

#### 8.4.9 status.factor_missing_ratio

当前样本中，feature.\* 列的缺失比例。

- 0.0：没有缺失。
- 0.2：20% 的 feature 缺失。
- 1.0：全部 feature 缺失。

它能表达因子不是只有有或无，还可能来了一半。对于 factor 表整行缺失的样本，此值填为 1.0。数据类型为 float32。

#### 8.4.10 status.feature_all_missing

当前样本的所有 feature.\* 是否全部为空。

- True：所有特征都缺失。
- False：至少有一个特征不缺失。

它通常和 factor_missing_ratio == 1.0 一致，但单独保留更方便写规则。

#### 8.4.11 status.sample_state

当前样本在 panel_base 层面的整体综合状态。这是整个 panel_base 最直观的总状态标签。枚举值如下。

- NORMAL：市场和因子都正常。
- SUSPENDED：停牌（无 OHLCV 记录）。注意停牌样本可能有因子数据。
- MARKET_ONLY：只有 OHLCV，没有 factor。
- FACTOR_ONLY：只有 factor，没有 OHLCV（通常与 SUSPENDED 重叠，优先标记为 SUSPENDED）。
- NO_SOURCE_RECORD：两边都没有（停牌且无因子数据）。
- PARTIAL_FACTOR_MISSING：因子部分缺失。
- INVALID_BASE_SAMPLE：从底层事实看就是异常样本。

判定优先级（从高到低）：NO_SOURCE_RECORD > SUSPENDED > MARKET_ONLY > PARTIAL_FACTOR_MISSING > NORMAL。

#### 8.4.12 status.sample_usable_for_feature

从 panel_base 当前事实来看，这条样本是否可以作为特征输入样本。

- True：可以作为 feature 输入。
- False：不可以作为 feature 输入。

判定依据为：has_factor_record=True 且 feature_all_missing=False。即这条样本能提供一个靠谱的特征向量 X。

这是 panel_base 中唯一的可用性标志。label 可用性和训练可用性不在 panel_base 中表达（详见 §8.6）。

### 8.5 来源存在性与缺失的区分规则

系统必须区分以下两类缺失。

1.  **row missing**：该 (date, sid) 在源表中整行不存在。  
    
2.  **field missing**：该行存在，但某些字段为空。

因此：

- status.has_market_record=False 表示 OHLCV 表整行不存在。
- status.bar_missing=True 可以表示行情字段无法使用，但不一定等同于整行不存在。
- status.has_factor_record=False 表示因子表整行不存在。
- status.feature_all_missing=True 表示因子行存在但所有 feature 都为空，或者 join 后所有 feature 均为空。
- status.factor_row_missing=True 专门用于表达因子表整行缺失。

### 8.6 usable flags 与分层职责

panel_base 只保留一个可用性标志：

- status.sample_usable_for_feature：是否可以作为 view 层特征输入来源。判定依据为 has_factor_record=True 且 feature_all_missing=False。

以下两个可用性判定不在 panel_base 中表达，而是由后续阶段动态生成：

- label 可用性：由 target_block 的 label_valid.\<label_name\> 表达，因为不同 label 的有效条件不同。
- 训练可用性：由训练阶段动态生成 train_mask，因为它依赖当前 label、model、objective 和 group 规则。

典型训练 mask 生成方式：

train_mask = (
    status.sample_usable_for_feature
    & label_valid.\<current_label\>
    & other_training_checks
)

### 8.7 停牌检测规则

在本系统使用的 OHLCV 数据源中，停牌股票在停牌日不会出现在 OHLCV 表中（整行缺失）。因此停牌检测逻辑为：

- 若一只股票在某交易日处于上市窗口内（在 active_session_index 中），但该 (date, sid) 在 daily_bars 中不存在，则判定为停牌。
- 即：status.is_suspended = status.is_listed AND NOT status.has_market_record。

此规则不依赖成交量为零的判断，因为停牌日根本没有 OHLCV 记录。

## 9\. Model-Label Contract 设计

系统必须显式支持两类模型：通用模型与强约束模型。通用模型可以适配任意合法 label；强约束模型会依赖 label 数量、角色、类型或具体名称，因此必须有训练前校验机制。

### 9.1 目标

- 避免模型内部悄悄依赖 config 中错误的 label。
- 让通用模型与强约束模型共用同一套训练入口。
- 让错误在 preflight 阶段就失败，而不是在模型 fit 时才失败。
- 为多任务训练提供清晰的标签角色表达。

### 9.2 建议的数据结构

@dataclass  
class TargetSpec:  
name: str  
task_type: str # regression / classification / ranking  
family: str | None  
horizon: int | None  
dtype: str = "float32"  
<br/>@dataclass  
class ModelLabelContract:  
mode: str # ANY_SINGLE / ANY_MULTI / EXACT / ROLE_BASED / CUSTOM  
min_labels: int = 1  
max_labels: int | None = 1  
exact_labels: list\[str\] = field(default_factory=list)  
required_roles: dict\[str, str\] = field(default_factory=dict)  
custom_validator: Callable | None = None

### 9.3 五种 contract 模式

| 模式  | 适用对象 | 说明  |
| --- | --- | --- |
| ANY_SINGLE | 通用单任务模型 | 接受任意一个合法 label。 |
| ANY_MULTI | 通用多任务模型 | 接受任意多个 label，但不要求固定名字。 |
| EXACT | 强绑定模型 | 必须使用指定 label 列表。 |
| ROLE_BASED | 推荐的多任务模式 | 模型依赖角色而不是具体 label 名称，例如 short、mid、long。 |
| CUSTOM | 复杂规则模型 | 通过自定义校验器表达更复杂的约束。 |

### 9.4 推荐做法

优先让强约束模型依赖角色，而不是依赖硬编码 label 名称。

```yaml
model:
  name: "lstm_mtl"
  label_roles:
    regression: "return_c0c1"
    classification: "momentum_cls"
```

训练前，training/preflight.py 读取 model contract 与 target specs，验证 label 数量、角色、task_type、family、horizon 等信息。验证失败时抛出清晰异常。

### 10.2 Refit 相关配置字段

训练配置中支持以下 refit 相关字段，控制模型选择后是否在更大数据集上重训练再评估测试集。

| 字段 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| run_test | bool | true | 是否在 test 集上做最终评估。设为 false 时只做 train+valid 阶段的选择。 |
| refit_before_test | bool | false | 是否在测试集评估前用 train+valid 数据 refit 模型。仅当 run_test=true 时有效。 |
| refit_train_scope | str | "train+valid" | refit 使用的数据范围。默认为 train+valid。 |
| refit_iteration_source | str | "selection" | refit 的迭代次数来源。"selection" 表示使用 selection 阶段在 valid 上确定的最佳迭代次数。 |

完整训练流程如下：

1. **Split**：按 valid_start、test_start 切分出 train / valid / test 三段，输出 SplitBundle（含 train_plus_valid_mask）。
2. **Train on train**：在 train 集上训练模型，以 valid 集作为 early stopping / evaluation 依据。
3. **Select on valid**：在 valid 集上评估，选出最优超参数与最佳迭代次数（best_iteration），冻结这些参数。
4. **Refit（可选）**：若 `refit_before_test=true`，用冻结的超参数和迭代次数，在 train+valid 上重新训练一个新模型。
5. **Evaluate on test**：若 `run_test=true`，在 test 集上评估最终模型（来自 refit 或 selection）。

关键约束：

- test 集在整个选择过程中不参与任何决策。
- refit 只能在 selection 完成之后进行，不能用 test 反馈来调整 refit 策略。
- refit 使用冻结的超参数和迭代次数，不重新做超参数搜索。
- 最终 test 评估结果必须记录模型来源（model_source: "selection" 或 "refit"）。
- 不开启 refit 时，整个流程与原来完全一致，无任何副作用。

### 11.2 Pipeline 执行与断点续跑

推荐使用 `run_pipeline.py` 一键运行全流程。该脚本支持断点续跑：

- 每步成功后写入检查点文件（`.pipeline_checkpoint.json`，位于项目根目录）

- 检查点记录 config_hash、已完成步骤列表、每步时间戳

# 全部从头跑（自动跳过已完成的步骤）

python scripts/run_pipeline.py --config configs/default.yaml --dry-run

1. **assets** — 构建基础数据
2. **panel_base** — 构建统一底板
3. **targets** — 构建标签
4. **view** — 构建模型视图（memmap）

## 13\. 实现边界与禁止事项

- 禁止在 panel_base 中直接存放 label.\* 列。
- 禁止让 view builder 反过来修改 panel_base 的 schema。
- 禁止在模型内部偷偷读取未经 contract 声明的 label 名称。
- 禁止把 active_session_index 当成某个 status 列。
- 禁止为了局部模型方便，破坏统一的列名前缀与索引规则。
- 禁止在没有 validator 的情况下默默容忍 schema 偏差。
- 禁止让 OHLCV 或因子表单独决定 panel_base 的行集合。
- 禁止把“整行缺失”和“字段缺失”混为一谈。
- 禁止在首版中引入过度复杂的依赖与框架。

## 18\. 补充输入数据约束

用户补充的硬约束如下：系统的标准输入统一只有两类数据，而且两类数据都以“每日每股”为粒度。实现时必须把这条约束视为最上游的数据接口定义。

硬约束如下。

- 标准输入一：每日每股 OHLCV 数据表。
- 标准输入二：每日每股因子数据表。
- 除非后续显式扩展，系统默认不接收分钟级数据、逐笔数据、事件流数据或其他非日频主输入。

### 18.1 输入数据定义

| 输入表 | 粒度  | 最低字段要求 | 说明  |
| --- | --- | --- | --- |
| daily_stock_ohlcv | daily × sid | date, symbol 或 sid, open, high, low, close, volume, amount | 这是市场行情主输入。 |
| daily_stock_factors | daily × sid | date, symbol 或 sid, factor_\* | 这是因子主输入；允许存在多个 feature 列。 |

### 18.2 对架构的直接影响

- sources/ohlcv.py 只负责读取每日每股 OHLCV 数据表。
- sources/factors.py 只负责读取每日每股因子数据表。
- assets 层不需要处理更复杂的多源输入融合，首版只围绕这两张表构建资产层。
- instrument_master、trading_calendar、daily_bars 主要从 OHLCV 表推导。
- factor_values 主要从因子表清洗与标准化得到。
- panel_base 的 market.\* 来自 daily_bars，feature.\* 来自 factor_values。
- 如果将来扩展分钟级或事件级输入，必须通过新增 source 与 asset，而不能破坏现有日频 contract。

### 18.3 直接实现要求

- 把每日每股 OHLCV 表和每日每股因子表视为唯一标准入口。
- 优先支持长表格式，必要时可兼容宽表，但最终都要标准化为 date、sid 为主键的长表逻辑。
- 如果原始输入只有 symbol，没有 sid，首版允许 sid == symbol。
- 任何样本构建、target 计算、view 构建与回测，都默认建立在日频样本上。
- 原始输入允许存在行不对齐。系统必须通过 active_session_index + left join + status flags 的方式统一表达这类不对齐，而不是在上游直接删行。
